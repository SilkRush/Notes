What is SQL?

Basic

SQL stands for Structured Query Language. Think of it as the language you use to talk to a database. You ask questions ("give me all users who signed up this month"), and the database answers. You also use it to store, change, and delete data.

🗃️ The Filing Cabinet Analogy

Imagine a huge filing cabinet. Each drawer is a database. Each folder inside is a table. Each sheet of paper is a row of data. SQL is how you search through, add, or remove those papers without opening every drawer yourself.

⚡ What SQL Can Do

SQL handles four main things: reading data (SELECT), writing data (INSERT), changing data (UPDATE), and deleting data (DELETE). These four together are called CRUD — Create, Read, Update, Delete.

🌍 Why "Universal"?

SQL is standardized, meaning MySQL, PostgreSQL, SQLite, MS SQL Server, Oracle — all of them speak SQL. The syntax is mostly the same everywhere. Small differences exist at the edges (a function name here, a syntax variant there), but 90% of what you learn works on any database.

💡

SQL is pronounced either "S-Q-L" (saying each letter) or "sequel" — both are completely accepted. Don't let anyone tell you one is wrong.

DROP TABLE IF EXISTS employees;

04

SELECT — Reading Data

Basic

SELECT is the most used SQL command by far. Every time you want to read or retrieve data, you use SELECT. It doesn't change anything — it just asks the database a question and shows you the answer.

How a SELECT query executes — click Run
SQL — SELECT Syntax
-- Select all columns
SELECT * FROM employees;

-- Select specific columns
SELECT name, department, salary FROM employees;

-- Give a column a nickname (alias)
SELECT name AS employee_name,
       salary * 12 AS annual_salary
FROM employees;

-- Select unique values only
SELECT DISTINCT department FROM employees;
⚠️

SELECT * is fine for exploring but avoid it in production code. It fetches every column even ones you don't need, which slows things down. Always name the columns you actually want.

05

WHERE — Filtering Rows

WHERE lets you filter rows. Instead of getting everything, you set conditions and only get the rows that match. It's like a gatekeeper — rows that don't meet the condition don't get through.

Interactive Filter — adjust and click Run Query
SELECT * FROM employees WHERE ...
SQL — WHERE operators
-- Comparison operators
WHERE salary > 80000
WHERE department = 'Engineering'
WHERE salary != 68000

-- Combine conditions
WHERE department = 'Engineering' AND salary > 90000
WHERE department = 'HR' OR department = 'Marketing'

-- Range check
WHERE salary BETWEEN 60000 AND 90000

-- Match a list
WHERE department IN ('Engineering', 'HR')

-- Pattern matching (% = any characters, _ = one char)
WHERE name LIKE 'A%'    -- names starting with A
WHERE name LIKE '%Chen'  -- names ending with Chen

-- Check for missing values
WHERE department IS NULL
WHERE department IS NOT NULL
06

ORDER BY & LIMIT

ORDER BY sorts your results. LIMIT cuts the result set to a specific number of rows. Together they're incredibly useful — like getting the top 5 highest-paid employees.

↕️ Sorting
SQL
-- Ascending (default)
SELECT name, salary
FROM employees
ORDER BY salary ASC;

-- Descending (highest first)
SELECT name, salary
FROM employees
ORDER BY salary DESC;

-- Sort by multiple columns
ORDER BY department ASC, salary DESC;
✂️ Limiting Results
SQL
-- Top 3 highest paid
SELECT name, salary
FROM employees
ORDER BY salary DESC
LIMIT 3;

-- Skip first 2, then get next 3
-- (pagination — page 2)
LIMIT 3 OFFSET 2;

07

INSERT / UPDATE / DELETE

These three commands let you modify the data in a table. INSERT adds rows, UPDATE changes existing ones, and DELETE removes them.

SQL — INSERT
-- Add a single row
INSERT INTO employees (id, name, department, salary, hire_date)
VALUES (7, 'Grace Tan', 'Engineering', 96000, '2024-01-08');

-- Insert multiple rows at once
INSERT INTO employees (id, name, department, salary, hire_date)
VALUES
  (8, 'Henry Park', 'HR', 70000, '2024-02-01'),
  (9, 'Iris Lam', 'Marketing', 67000, '2024-03-15');
SQL — UPDATE
-- Give Alice a raise
UPDATE employees
SET salary = 98000
WHERE id = 1;

-- Update multiple columns at once
UPDATE employees
SET salary = salary * 1.10, -- 10% raise
    department = 'Senior Engineering'
WHERE department = 'Engineering' AND salary > 90000;
⚠️

Always write the WHERE clause first when doing UPDATE or DELETE. If you forget it, you'll change or delete every single row in the table. No undo. Test your WHERE with a SELECT first before running the UPDATE/DELETE.

SQL — DELETE
-- Delete one specific row
DELETE FROM employees
WHERE id = 5;

-- Delete all employees in a department
DELETE FROM employees
WHERE department = 'Marketing';

-- TRUNCATE: faster way to delete ALL rows (no WHERE)
TRUNCATE TABLE employees;

Aggregate Functions

Intermediate

Aggregate functions collapse multiple rows into a single calculated value. Instead of seeing all 6 salaries, you might want just the average salary. That's what these do.

COUNT() SUM() AVG() MAX() MIN()
Salary aggregates by department — click a function
SQL — Aggregate examples
-- Total number of employees
SELECT COUNT(*) AS total_employees FROM employees;

-- Average salary across the whole company
SELECT AVG(salary) AS avg_salary FROM employees;

-- Highest salary in Engineering
SELECT MAX(salary)
FROM employees
WHERE department = 'Engineering';

-- Total payroll budget
SELECT SUM(salary) AS total_payroll FROM employees;

-- COUNT ignores NULL values
SELECT COUNT(department) FROM employees; -- skips NULLs
SELECT COUNT(*) FROM employees;       -- counts all rows
09

GROUP BY & HAVING

GROUP BY takes the rows, splits them into groups by some column, then runs an aggregate function on each group. HAVING is like WHERE but it filters after grouping — it filters groups, not rows.

💡

Key rule: Every column in your SELECT that isn't inside an aggregate function must appear in the GROUP BY clause. If you're selecting name, department, and AVG(salary) — department goes in GROUP BY, name probably shouldn't be there at all.

SQL — GROUP BY & HAVING
-- Average salary per department
SELECT department, AVG(salary) AS avg_salary
FROM employees
GROUP BY department;

-- How many employees in each department?
SELECT department, COUNT(*) AS headcount
FROM employees
GROUP BY department
ORDER BY headcount DESC;

-- Only departments with avg salary above 80k
-- WHERE filters rows BEFORE grouping
-- HAVING filters groups AFTER grouping
SELECT department, AVG(salary) AS avg_sal
FROM employees
GROUP BY department
HAVING AVG(salary) > 80000;
📋 SQL Execution Order (Important!)

SQL doesn't execute left to right. The actual order is:

FROM — pick which table(s)
WHERE — filter individual rows
GROUP BY — bundle rows into groups
HAVING — filter whole groups
SELECT — pick which columns to show
ORDER BY — sort the results
LIMIT — cut to N rows

JOINs — Combining Tables

In a real database, data is split across multiple tables to avoid repetition. JOINs let you combine them based on a shared column. This is one of the most important SQL concepts.

JOIN Types Visualizer — click each join
employees
departments
Returns only rows where there's a match in BOTH tables. Unmatched rows are excluded.
SQL — All JOIN types with example tables
-- Setup: employees has dept_id, departments has id + dept_name

-- INNER JOIN: only matched rows
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;

-- LEFT JOIN: all employees + their dept (NULL if no dept)
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;

-- RIGHT JOIN: all departments + employees (NULL if no employees)
SELECT e.name, d.dept_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.id;

-- FULL OUTER JOIN: everything from both tables
SELECT e.name, d.dept_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;

-- Join 3 tables
SELECT e.name, d.dept_name, p.project_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id
INNER JOIN projects p ON p.dept_id = d.id;
✅

Using table aliases (e, d, p) makes long queries much cleaner. Write FROM employees e and then use e.column_name everywhere. Much less typing, much easier to read.

Subqueries

A subquery is a query nested inside another query. The database runs the inner query first, then uses that result in the outer query. It's like asking "give me employees whose salary is above the result of this other calculation."

Click the inner query to see what it returns
SELECT name, salary FROM employees
WHERE salary > ( SELECT AVG(salary) FROM employees )
→ Returns a single value: 81,500 (average salary)
The outer query then filters: WHERE salary > 81500
SQL — Subquery types
-- Scalar subquery: returns one value
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- IN subquery: returns a list
SELECT name FROM employees
WHERE dept_id IN (
  SELECT id FROM departments
  WHERE location = 'New York'
);

-- EXISTS subquery: checks if rows exist
SELECT name FROM employees e
WHERE EXISTS (
  SELECT 1 FROM projects p
  WHERE p.lead_id = e.id
);

-- Subquery in FROM (derived table)
SELECT dept, avg_sal
FROM (
  SELECT department AS dept, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY department
) AS dept_averages
WHERE avg_sal > 70000;

Window Functions

Advanced

Window functions are like aggregate functions but they don't collapse rows. You get one result per row, but the calculation uses a "window" of surrounding rows. The OVER() clause defines the window.

💡

The big difference: GROUP BY + AVG() gives you one row per group. AVG() OVER() gives you the average next to every row. All rows stay — you just get extra calculated columns alongside them.

Window function results — click a function to highlight
name dept salary rank_in_dept
SQL — Window functions
-- Rank employees by salary within their department
SELECT
  name, department, salary,
  RANK() OVER (
    PARTITION BY department
    ORDER BY salary DESC
  ) AS rank_in_dept
FROM employees;

-- ROW_NUMBER: unique number per row (no ties)
-- RANK: ties share same rank, next rank skips
-- DENSE_RANK: ties share same rank, no gaps

-- Running total of salary
SELECT name, salary,
  SUM(salary) OVER (ORDER BY id) AS running_total
FROM employees;

-- Compare to previous row (previous employee's salary)
SELECT name, salary,
  LAG(salary, 1) OVER (ORDER BY salary) AS prev_salary,
  LEAD(salary, 1) OVER (ORDER BY salary) AS next_salary
FROM employees;

-- Average salary within each department (keeps all rows)
SELECT name, salary,
  AVG(salary) OVER (PARTITION BY department) AS dept_avg
FROM employees;

CTEs — WITH clause

CTE stands for Common Table Expression. It's a named temporary result set you define at the top of your query using WITH, then reference by name. It makes complex queries dramatically easier to read and write. Think of it as giving a subquery a proper name.

CTE execution flow
dept_averages
AVG salary per dept
→
top_depts
Departments above 80k avg
→
Final SELECT
Employees in top depts
SQL — CTEs
-- Single CTE
WITH dept_averages AS (
  SELECT department, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY department
)
SELECT e.name, e.salary, d.avg_sal
FROM employees e
JOIN dept_averages d ON e.department = d.department
WHERE e.salary > d.avg_sal;

-- Multiple CTEs (chain them with a comma)
WITH
dept_avgs AS (
  SELECT department, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY department
),
top_depts AS (
  SELECT department
  FROM dept_avgs
  WHERE avg_sal > 80000
)
SELECT e.name, e.salary
FROM employees e
WHERE e.department IN (SELECT department FROM top_depts);
✅

CTEs don't make queries faster — they make them readable. Use them whenever a subquery is getting complex or you find yourself writing the same subquery twice. At a certain level of complexity, CTEs are the difference between code anyone can understand and code only you can.

Indexes

Advanced

An index is a separate data structure the database maintains to speed up lookups. Without an index, every query scans every row in the table (full table scan). With an index on the column you're searching, the database jumps directly to the right rows.

📚 The Book Analogy

Imagine a 500-page textbook. Finding the word "recursion" without an index means reading every page. With an index at the back of the book, you jump straight to page 312. Same idea. Indexes are a sorted lookup structure — typically a B-tree — maintained automatically by the database.

Without Index
With Index
SQL — Creating Indexes
-- Basic index on a column you frequently filter by
CREATE INDEX idx_department
ON employees (department);

-- Unique index: enforces uniqueness + speeds up lookup
CREATE UNIQUE INDEX idx_email
ON employees (email);

-- Composite index: speeds up queries filtering by both
CREATE INDEX idx_dept_salary
ON employees (department, salary);

-- Drop an index
DROP INDEX idx_department;
⚠️

More indexes isn't always better. Each index takes storage space and slows down INSERT/UPDATE/DELETE (the database has to update the index too). Index columns you actually filter or join on. Don't index everything blindly.

15

Views

A view is a saved SQL query that you can treat like a table. Every time you query the view, it runs the underlying query and returns fresh results. Views don't store data — they store the query definition.

🔭 Why Use Views?

They simplify complex queries so other people (or your future self) don't have to remember all the joins and conditions. They also add a security layer — you can let someone query a view without giving them access to the raw table.

⚡ View vs CTE

A CTE exists only within one query — gone when the query ends. A view is saved permanently in the database and reusable across many different queries and sessions.

SQL — Views
-- Create a view
CREATE VIEW engineering_team AS
SELECT name, salary, hire_date
FROM employees
WHERE department = 'Engineering';

-- Now query it just like a table
SELECT * FROM engineering_team
WHERE salary > 90000;

-- Update a view definition
CREATE OR REPLACE VIEW engineering_team AS
SELECT name, salary, hire_date, id
FROM employees
WHERE department = 'Engineering';

-- Remove a view
DROP VIEW engineering_team;

Transactions & ACID

Advanced

A transaction is a group of SQL operations that either all succeed or all fail together. Transactions protect your data from ending up in a broken half-updated state — especially critical when multiple operations need to happen as one atomic action.

A
Atomicity

All operations in the transaction succeed, or none of them do. No partial results. If step 3 of 5 fails, steps 1 and 2 get rolled back.

C
Consistency

The database always moves from one valid state to another. No constraints get violated. The data stays meaningful before and after.

I
Isolation

Concurrent transactions don't interfere with each other. One transaction doesn't see another's incomplete changes.

D
Durability

Once committed, the changes are permanent. Even if the server crashes right after, the committed data survives.

BEGIN;  -- or BEGIN TRANSACTION; or START TRANSACTION;

  UPDATE accounts
  SET balance = balance - 500
  WHERE id = 1;

  UPDATE accounts
  SET balance = balance + 500
  WHERE id = 2;

COMMIT;  -- save everything permanently

-- If something goes wrong, undo all changes
ROLLBACK;
17

Query Optimization

Advanced

As tables grow to millions of rows, poorly written queries grind to a halt. Optimization is about knowing why a query is slow and how to fix it without changing what it returns.

SQL — EXPLAIN: see how the database executes your query
-- EXPLAIN shows the query plan without running it
EXPLAIN
SELECT name, salary
FROM employees
WHERE department = 'Engineering';

-- EXPLAIN ANALYZE actually runs it and shows timing
EXPLAIN ANALYZE
SELECT * FROM employees
WHERE salary BETWEEN 80000 AND 100000;
🚫 What Kills Performance

Filtering on unindexed columns in large tables. Using functions on indexed columns in WHERE (WHERE LOWER(name) = ... kills the index). Using SELECT * when you only need 2 columns. Fetching millions of rows with no LIMIT when you only need a few. Correlated subqueries that run once per row.

✅ What Helps

Add indexes on columns used in WHERE, JOIN ON, and ORDER BY. Only SELECT the columns you actually need. Use CTEs to break complex queries apart. Use JOINs instead of correlated subqueries when possible. Filter early — reduce row count as soon as possible in the query.

SQL — Anti-patterns & fixes
-- BAD: function on column kills index usage
WHERE YEAR(hire_date) = 2021

-- GOOD: use range comparison instead
WHERE hire_date BETWEEN '2021-01-01' AND '2021-12-31'

-- BAD: NOT IN with subquery can behave oddly with NULLs
WHERE id NOT IN (SELECT manager_id FROM employees)

-- GOOD: use NOT EXISTS instead
WHERE NOT EXISTS (SELECT 1 FROM employees e2 WHERE e2.manager_id = e.id)

-- BAD: SELECT * pulls all 30 columns
SELECT * FROM employees

-- GOOD: only the 2 columns you need
SELECT name, salary FROM employees
